# AGENTS

## Core Principle

topics 目录是一个持续增量编译的知识库。每次回答都应优先复用已有知识，而不是从零推导；每次有价值的新结论都应回流到知识库，使其持续增值。最终裁决优先级始终是：**当前代码 > 官方文档 > topic 文档**。设计理念详见 `llm-wiki.md`。

## Context

用户主要关注 `vllm + vllm-ascend + xLLM` 在 NPU（Ascend）上的逻辑。

- `vllm-ascend`：NPU 芯片相关代码从 vLLM 上游解耦出来的插件仓库。
- `xLLM`：通过 `import vllm` + `import vllm_ascend` 在 NPU 上运行。
- 代码仓库位于 `/Users/bytedance/gitspace`。

## Layer 1: Answer Flow（最高优先级）

回答解释类问题时，按以下顺序执行：

1. **读 `topics/index.md`** —— 通过关键词和摘要快速定位候选 topic。
2. **命中** → 读对应 topic，获取已有结论、代码链路和易混淆点，加速理解。
   **未命中** → 直接进入代码和官方文档，从头调研。调研完成后按 Layer 2 判断是否值得新建 topic。
3. **回查代码和官方文档核验** —— 无论是否命中 topic，最终都必须回到代码核验。如果 topic 与代码不一致，以代码为准。
4. **回答用户** —— 先给简洁总结，再展开细节。

## Layer 2: Knowledge Maintenance（回答后按需执行）

回答完成后，判断是否需要维护 topic 文档。不是每次都要写，按需判断。

### 默认复用，按需更新

- 已有 topic 能覆盖 → 直接复用，不新建。
- 已有 topic 部分覆盖但主题一致 → 更新已有 topic，补充或修正。
- 仅措辞不同、局部展开 → 不新建。

### 新建门槛

同时满足以下三条才新建：

1. 当前问题对应一个明确的新主题；
2. 现有 topic 无法自然归入；
3. 新文档能为后续类似问题提供稳定复用价值。

### 合并与删除

- 多篇文档解释同一条主链路或大量重复 → 合并进总览文档。
- 被新文档完整吸收的旧文档 → 删除（确认核心信息已迁移、不丢失高频独立主题）。

### 维护后必做

- **更新 `topics/index.md`**：同步关键词和摘要。
- **验收**：结论可追溯到代码或官方文档，无过时信息，无绝对路径。

## Layer 3: Writing Standards（参考级）

### 结构

代码解释类 topic 默认包含：`结论速览` / `代码链路` / `易混淆点`。至少保证有一个高层结论区和一个可追溯依据区。

### 头部元数据

每个 topic 文件顶部应包含 tags 行，便于 index 检索：

```
---
tags: [关键词1, 关键词2, ...]
---
```

### 命名

- 文件名 `snake_case`，优先"主题总览式"命名。
- 如果是某个总览文档的子问题，优先更新总览文档。

### 路径与引用

- 不写绝对路径，不用 `file://` 链接。
- 统一使用仓库内相对路径，例如 `vllm/vllm/config/parallel.py`。

### 内容

- 结论落到真实代码符号和实现链路。
- 解释"是什么、为什么、在哪条链路上发生"。
- 写稳定、可复用的结论，不记录一次性对话上下文。

## Periodic Consolidation

仅在以下场景触发系统整理（不是每次答题的默认动作）：

- 用户明确要求整理；
- topic 数量明显膨胀或重复率升高；
- 现有结构妨碍检索和复用。

整理时：先对照代码确认结论成立，再合并同类文档，删除已被吸收的旧文档，最后更新 `index.md`。
