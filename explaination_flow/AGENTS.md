## Context Related Rules
你需要按真实情况回答问题。

在大多数情况下，用户会提问 `vllm`、`vllm-ascend` 和 `xLLM` 这三个代码库的问题。当问题涉及这三个仓库时，你必须基于这三个仓库当前的实际代码和文档来回答，不要凭空猜测，也不要把历史印象当成当前事实。

我主要关注的是 `vllm + vllm-ascend + xLLM` 在 NPU（Ascend）芯片上的逻辑部分。

- `vllm-ascend` 是把 NPU 芯片相关代码从 vLLM 上游仓库中解耦出来的插件仓库。
- `xLLM` 需要通过 `import vllm`、`import vllm_ascend` 才能在 NPU 芯片上运行。
- 一个核心目标是让 `xLLM` 能在 NPU 上跑通。

这些代码仓库都位于 `/Users/bytedance/gitspace`。

## Reference Local Topic Documents for Similar Questions
在回答解释类问题时，先检索本地 topic 文档库，再回查代码和官方文档核验。

执行顺序应为：

1. 先查看 `tools/explaination_flow/topics` 中是否已经有相同或相近主题的总结文档。
2. 如果已有 topic 能覆盖当前问题，先用它帮助你快速定位关键链路、术语和已有结论。
3. 无论 topic 是否命中，只要问题涉及代码行为、配置语义、运行链路或版本相关差异，最终都必须回到代码和官方文档核验。

注意：

- topic 文档的作用是帮助你加速理解和检索，不是最终真相源。
- 最终裁决优先级始终是：当前代码和官方文档，高于 topic 文档。
- 如果 topic 文档和代码不一致，必须以代码和官方文档为准，并在必要时修正文档。

## Update Existing Topics First
默认优先复用已有 topic，按需更新，不要持续新增细粒度文档。你需要采取增量，优化式方案来壮大文档，实现复利，以供后续可以快速定位。这里的总规则请遵守`llm-wiki.md`。

处理原则如下：

1. 如果已有 topic 已经能准确覆盖当前问题，默认直接复用，不强制新建或改写文档。
2. 如果已有 topic 只能部分覆盖，但主主题一致，优先更新已有 topic，补充缺失结论或修正过时内容。
3. 只有当当前问题属于一个明确的新主题，且现有 topic 无法自然吸收时，才创建新文档。
4. 普通重复问答、仅措辞不同的问题、或只是对已有 topic 的局部展开，默认不新建 topic。
5. 每次更新过后的验收标准：检查文档的结论是否与代码库的当前状态一致，是否能追溯到代码或官方文档。

## Topic Lifecycle Rules
`tools/explaination_flow/topics` 中的 topic 文档有四种维护动作：新建、更新、合并、删除。

### 新建
仅在以下条件同时满足时新建：

- 当前问题对应一个明确的新主题；
- 现有 topic 无法自然归入；
- 新文档能为后续类似问题提供稳定复用价值。

### 更新
出现以下情况时优先更新已有文档：

- 代码行为变了，原结论过时；
- 原文档覆盖不完整，但主题仍一致；
- 当前问题只是已有总览文档中的一个子问题。

### 合并
出现以下情况时应考虑合并 topic：

- 多篇文档解释的是同一条主链路或同一个大类问题；
- 不同小文档之间大量重复；
- 现有主题已经更适合放进一个总览文档而不是继续拆分。

默认优先把新信息合并进已有总览文档，而不是继续新增细粒度 topic。

### 删除
如果某篇 topic 已经被新的总览文档完整吸收，且继续保留只会造成重复或混淆，可以删除。

删除前应确认：

- 被删文档的核心信息已被吸收到新文档中；
- 不会丢失仍然高频使用的独立主题；
- 目录结构会因此更清晰，而不是更混乱。

## Topic Writing Rules
topic 文档是给后续 AI agents 和用户快速复用的说明材料，要求准确、短平快、可检索。

### 结构要求
代码解释类 topic 文档默认应包含以下三部分：

1. `结论速览`
2. `代码链路`
3. `易混淆点` 或 `场景说明`

如果主题天然不适合这三段，也应保证至少有：

- 一个高层结论区
- 一个可追溯的依据区

### 命名要求
- 文件名使用 `snake_case`。
- 优先使用“主题总览式”命名，而不是一次性问句式命名。
- 如果当前主题本质上是某个总览文档下的子问题，优先更新总览文档，不要单独造一个新文件名。

### 路径与引用要求
- 在 topic 文档内容中，不要写绝对路径。
- 文件路径统一写对应仓库内的相对路径，例如：
  - `vllm/vllm/config/parallel.py`
  - `vllm-ascend/vllm_ascend/worker/model_runner_v1.py`
  - `xLLM/xllm/backend/distributed.py`
- 不要在 topic 文档中使用 `file://` 链接。

### 内容要求
- 关键结论必须尽量落到真实代码符号、实现链路或官方文档位置。
- 尽量解释“是什么、为什么、在哪条链路上发生”，而不是只堆路径名。
- 优先写稳定、可复用的结论，不要记录只服务于一次对话的零碎上下文。

## Summary Requirement
在回答解释类问题后，你应先给用户一个简洁总结，再判断是否需要同步维护 topic 文档。

这里的“需要维护”是按需判断，不是每次回答后都强制写 topic。

## Periodic Consolidation
topic 库的整理与合并是独立维护任务，不是每次答题后的默认动作。

只有在以下场景下，才应触发对 `tools/explaination_flow/topics` 的系统整理：

- 用户明确要求整理、重构或归并 topic 库；
- topic 数量明显膨胀；
- 同类文档重复率明显升高；
- 现有 topic 结构已经妨碍检索和复用。

整理时遵守以下规则：

1. 先对照代码库和官方文档，确认文档结论仍然成立。
2. 将属于同一个大类、同一条主链路、或同一个总览主题的文档合并。
3. 被新文档完整吸收的旧文档可以删除。
4. 合并后的文档需要结构清晰，优先形成少量高质量总览文档，而不是保留大量碎片化小文档。

## Acceptance Checklist
无论是新建、更新、还是整理 topic，最后都应至少检查以下内容：

1. 文档中的关键结论是否能追溯到当前代码或官方文档。
2. 文档结论是否与当前代码行为一致，没有明显过时信息。
3. 文档中是否残留绝对路径或 `file://`。
4. 当前改动是否应合并进已有总览文档，而不是继续新增碎片化 topic。
5. 如果你更新了某个总览文档，是否存在应被吸收或删除的旧文档没有处理。
